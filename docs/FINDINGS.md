# 실전 면접 성능 조사 — 결론

측정 환경: 로컬, RTX 3070 Ti Laptop(8GB), MySQL 8, uvicorn 단일 워커. (2026-07)

## 한 줄 요약

실전 면접의 병목은 **SenseVoice STT 하나**다. VAD를 두 번 돌리던 낭비를 한 번으로 줄여
처리량을 +35% 올렸다(적용·병합 완료). 남은 동시성 한계는 **단일 GPU의 구조적 직렬화**이고,
스레드·프로세스·배치 어느 것으로도 못 넘는다 — 수평 확장(GPU 추가)이 답이다.

시나리오로 잰 실제 용량(GPU 1장): **평균 동시 15명 / 최악(동시 시작) 5명 미만.** 저사양
기기가 JPEG 를 보내 서버 MediaPipe 를 태우면 각각 ~10명 / 3명 미만으로 떨어진다(스레드
피크 768개). 무너지는 지점은 언제나 `/answer`(STT) 이고, 시작 시각이 겹칠수록 낮아진다.

## 1. 병목은 STT다 (다른 후보는 전부 기각)

실사용 페이스(준비 10초 / 답변 75초)로 동시 면접을 올렸을 때:

| 동시 면접자 | `/answer` p95 | 판정 (예산 10초) |
|---|---|---|
| 3명 | 8.57s | 통과 |
| 4명 | 10.36s | 실패 |
| 5명 | 14.47s | 실패 |

`/answer` 말고는 아무것도 안 무너졌다 (5xx 0, 커넥션 풀 고갈 0, 영상 프레임 왕복 15ms,
백그라운드 분석 수렴 15.8초). 기각된 가설:

- **H1 커넥션 풀 고갈**: 0건. 영상 소켓은 SQLAlchemy Session을 첫 쿼리에서야 잡는데
  프레임 스트리밍 중엔 쿼리가 없다.
- **H3 백그라운드 분석 적체**: 5명에서도 15.8초면 수렴.
- **외부 LLM**: 동시 5명에서도 다음 질문 생성 평균 2.0초로 일정 — 병목 아님.

`/answer` 지연의 대부분은 **STT 대기**였다.

## 2. STT를 고쳤다 — VAD 두 번 → 한 번 (적용·병합 완료)

`sensevoice.py`가 같은 오디오에 VAD를 두 번 돌리고 있었다:

- `asr.generate` — 내부에서 VAD로 끊어 추론하고 합친다
- `vad.generate` — 말속도용 발화 구간을 얻으려고 같은 오디오에 VAD를 **또** 돌린다 (STT의 21%)

두 번째가 계산하는 구간은 `asr`가 내부에서 **이미 계산해 버리고 있었다.**
`asr.vad_model.inference`를 한 번 감싸(threading.local, 동시 추론 대비) 그 값을 주워오고,
별도 VAD 모델 로드와 두 번째 패스를 삭제했다.

> 참고: `output_timestamp=True`로 `generate`가 타임스탬프를 내게 하는 공식 경로도 재봤으나,
> CTC 강제정렬 때문에 `asr.generate`가 +0.72초 늘어 **오히려 손해**라 쓰지 않았다.

**결과 (앱 레포 `78/stt` → develop 병합):**

| 지표 | Before | After |
|---|---|---|
| 순수 STT 처리량 (동시 1) | 0.72 건/초 | **0.97 건/초 (+35%)** |
| 한 건 GPU 시간 | 1.38s | 1.03s |
| `/answer` p95 (동시 3, LLM 2.0초 고정) | 13.6s | **10.7s (-22%)** |
| fsmn-vad 로드 | 2회 | 1회 (기동 11→6.7초) |
| 말속도 발화 길이 | 기준 | +0.76% 차이 (내부 VAD의 30초 청킹) |

말속도 0.76% 차이는 채점 라벨·점수를 바꾸지 않음을 확인(경계 ±1음절/분에서만 영향).
단위 테스트(`test_sensevoice_stt.py` 등) 전부 통과.

## 3. 남은 동시성 한계 — 쉬운 지렛대가 전부 안 통한다

"한 건"을 줄인 뒤에도 동시 처리량은 ~0.9건/초에 묶여 있다. 원인을 실측으로 좁혔다.

### 한 건 CPU/GPU 분해 (`probe_gpu.py`)

| 구간 | 시간 |
|---|---|
| librosa.load (CPU) | 218 ms |
| generate wall | 1085 ms |
| └ GPU 커널 활성 | 687 ms |
| └ CPU self | 964 ms |

CPU 일이 GPU 일보다 크다 → 처음엔 "GIL/CPU 병목" 가설을 세웠다.

### 2스레드 vs 2프로세스 (`exp_concurrency.py`)

| 방식 | 처리량 | 개별 latency | VRAM peak |
|---|---|---|---|
| 2스레드 (모델 1개 공유) | 0.89 건/초 | 2235 ms | 3748 MB |
| 2프로세스 (모델 각 1개) | 0.78 건/초 (-12%) | 2551 ms | 7126 MB |

**프로세스로 쪼갰는데 처리량이 안 늘었다 → GIL 가설 반증.** GIL·세마포어·모델을 공유하지
않는 두 프로세스가 그래도 직렬화된다는 건, 병목이 **둘이 공유하는 단 하나의 자원 = GPU**라는
뜻이다. 프로세스가 오히려 느린 건 MPS 없는 컨텍스트 전환 비용, VRAM은 모델 2개로 **거의 배**
(8GB 중 7.1GB — 3번째 프로세스 불가).

### 배치 1/2/4/8 (`exp_batch.py`)

| 배치 | 처리량 | GPU 사용률 |
|---|---|---|
| 1 | 0.97 건/초 | 75% |
| 2 | 0.94 건/초 | 68% |
| 4 | 0.90 건/초 | 64% |
| 8 | 0.79 건/초 (-18%) | 64% |

**배치도 반증.** FunASR의 `generate(input=[리스트])`는 GPU에서 묶지 않고 입력을 파이썬
루프로 하나씩 돈다. 그래서 묶을수록 CPU 구간만 길어져 GPU 사용률이 떨어진다.

## 최종 진단

| 지렛대 | 결과 | 이유 |
|---|---|---|
| 스레드 늘리기 | ✗ | 단일 스트림 직렬 |
| 프로세스 늘리기 | ✗ (-12%, VRAM 2배) | 단일 GPU가 공유 직렬 자원, MPS 없음 |
| 리스트 배치 | ✗ (-18%) | FunASR가 입력을 루프 |

- **단일 GPU에서 이 SenseVoice 파이프라인은 ~0.9~0.97 건/초가 실질 천장**이고, 그게 이
  모델·이 GPU의 한계에 가깝다.
- 연속 부하에서 GPU 사용률은 실제로 **75%** — "사용률이 낮다"는 건 요청 사이 짬 때문이지
  코어가 놀아서가 아니었다. (조사 초기의 오해)
- 남는 ~25%는 쉬운 knob으로 못 쓴다. 쓰려면 AutoModel 아래에서 여러 요청의 피처를 **진짜
  텐서 배치**로 묶어야 하는데, knob이 아니라 코드 작업이고 효과·리스크가 불확실하다.

## 권장

1. **~0.9건/초 = GPU 1장의 용량으로 받아들이고 수평 확장** (STT 워커를 GPU마다 하나씩).
   용량 산정이 명확해진다.
2. **한 건 자체를 더 빠르게** — fp16/양자화, 또는 더 가벼운 STT. (이번 VAD 중복 제거가
   이 계열이었고 1.38→1.03초로 한 발 갔다.)
3. 진짜 텐서 배칭은 효과가 검증되면 그때. 지금은 우선순위 낮음.

## 4. 시나리오 기반 용량 — "실제로 동시 몇 명?"

격리 벤치가 아니라 전체 면접 흐름(start → 질문마다 [영상 소켓 + 답변 업로드] → closing →
분석 대기)을 실제로 돌려 잰 용량. 실 STT, LLM 은 2.0초 고정(실측 평균), 답변 오디오는
길이별(30/60/90초)로 랜덤, 서버 자원(ffmpeg·임베딩·DB·영상 소켓) 전부 실제.

두 시나리오로 나눈다:
- **평균**: 사용자가 랜덤 시각에 도착(도착 지터), 답변 길이 제각각. 지속 가능한 용량.
- **최악**: 전원이 같은 순간(지터 0, 준비 0초) 가장 무거운 90초 답변을 제출. 순간 최대 압력.

| 시나리오 | 감당 한계 | 첫 병목 |
|---|---|---|
| **평균** | 동시 **15명** (10·15 통과, 20 실패) | STT (`/answer` p95) |
| **최악** | 동시 **5명 미만** (5명 p95 13.7초) | STT (`/answer` p95) |

두 경우 다 5xx·커넥션 풀·영상 소켓·분석 큐는 멀쩡(전부 0 또는 여유). 무너진 건 언제나
`/answer`(STT). **왜 3배 차이나냐** — STT 요청이 시간축에 어떻게 분포하느냐 하나다:
평균은 요청이 흩어져 GPU가 초당 ~0.9건을 꾸준히 소화하고, 최악은 N건이 한 순간에 몰려
직렬 STT 큐가 N배로 쌓인다(최악 5명 p95 13.7초 → 10명 28.4초, 거의 선형).

참고로 "터진다"는 서버 다운이 아니라 **`/answer` p95가 10초(준비 시간 예산)를 넘는다**는
뜻이다. 5xx 는 끝까지 0이었다.

**운영 함의**: 동시 사용자 수보다 **시작 시각이 겹치는지**가 용량을 좌우한다 →
입장 큐/슬롯으로 시작을 분산하는 게 인원 늘리기보다 효과가 크다.

## 5. 서버 MediaPipe(JPEG 폴백)를 켜면

기본은 브라우저가 랜드마크를 뽑아 보낸다(서버 CPU 마이크로초). 저사양 기기가 JPEG 프레임을
보내면 서버가 **접속마다 MediaPipe 그래프(FaceMesh+Pose)를 올려** 프레임당 ~18ms CPU 로
처리한다(`video.py` JPEG 폴백). 이 경로를 강제로 태워 봤다.

| 시나리오 | 랜드마크(기본) | 서버 MediaPipe(JPEG) |
|---|---|---|
| 평균 한계 | ~15명 | ~10~13명 (15명 p95 10.16초) |
| 최악 한계 | ~5명 미만 | **3명 미만** (3명 p95 10.96초) |

**스레드 폭증 (요청받아 측정):**
- 코드상 `asyncio.to_thread` 기본 풀 = **24개**(다음질문 LLM 과 공유), 동시성 제한 **없음**.
- 실측 피크: 동시 10명 **709 스레드**, 15명 **768 스레드**. 접속당 MediaPipe + OpenCV 가
  내부 스레드를 띄우는데 `cv2.setNumThreads` 제한이 없어 **20코어를 ~38배 오버서브스크립션**.
- 메모리: 동시 10명 **~5GB** (접속당 104MB × N + STT/임베딩 ~2GB).

**핵심**: MediaPipe 는 영상 경로를 막지 않는다(왕복 32~161ms 로 계속 정상). 무너지는 건
언제나 `/answer`(STT) — MediaPipe 의 **CPU·스레드 폭증이 STT 빠른 경로를 밀어내서**다.
평균에선 분산 덕에 타격이 작지만(10명까지 랜드마크와 거의 동일), 최악에선 전원의 그래프가
동시에 켜져 5명→3명 미만으로 떨어진다.

**완화책**: (a) `cv2.setNumThreads(1)` 로 접속당 스레드 폭증 억제, (b) MediaPipe 동시 실행에
세마포어(STT 처럼)로 스레드/메모리 상한. 근본적으론 브라우저 랜드마크 경로를 최대한 유지.

## 부록 — 조사 중 정정한 것들 (정직 기록)

측정이 가설을 여러 번 뒤집었다. 남겨 둔다:

1. "GPU가 직렬이라 못 늘린다" (처리량 정체만 보고 단정) → **부분만 맞음.** 칩이 포화라서가
   아니라 단일 GPU가 독립 작업을 겹쳐 못 돌리는 구조 때문.
2. "그럼 CPU/GIL이 병목이겠다" (프로파일러의 CPU>GPU만 보고) → **반증.** 프로세스 분리가
   처리량을 못 올렸다.
3. "배치로 GPU 여유를 쓰면 된다" → **반증.** FunASR는 입력을 배치 안 하고 루프.
4. "백그라운드 임베딩이 STT의 GPU를 뺏는다" → **반증.** `EMBED_DEVICE=cpu` A/B에서 차이
   없었다.
5. "LLM을 0초로 해도 `/answer`가 안 줄었다" → **잘못된 대조**(서로 다른 서버 인스턴스 비교).
   같은 서버에서 조건만 바꾸면 LLM 시간은 실제로 더해진다.

교훈: 사용률·처리량 같은 단일 지표로 병목의 **위치**를 단정하지 말 것. 분해(프로파일러)와
격리 실험(프로세스/배치)으로 좁혀야 한다.
